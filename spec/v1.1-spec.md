# YT Restore v1.1 Plan

## Scope

This release adds user-facing control over how overlay buttons behave and a way to fully rebuild the extension state when YouTube changes under us.

## Goals

1. Allow reversing the order of the overlay buttons.
2. Allow hiding custom overlay buttons when YouTube native buttons are already present.
3. Add a reset/rebuild action that clears extension state and re-discovers overlays from scratch.
4. Add a Donate link that points to the maintainer’s KoFi page.

## Assumptions

- The current extension architecture remains the same: `src/content.js` handles DOM discovery and injection, `src/inject.js` handles page-context menu actions, and `src/styles.css` owns the UI presentation.
- The extension does not currently persist any settings, so v1.1 will introduce a small persistence layer rather than reusing existing storage.
- The reset/rebuild action should clear extension-managed state and rebuild overlays immediately; it should not silently wipe unrelated browser data.
- The KoFi URL is not yet provided, so implementation should treat it as a required configuration value before release.

## Proposed UX

Add a lightweight extension settings surface, either as a popup or an options page, with these controls:

- Toggle: Reverse overlay button order.
- Toggle: Hide overlays when native YouTube buttons are present.
- Action: Rebuild overlays now.
- Link: Donate (https://ko-fi.com/captaincodeuk).

If a popup is not desired, the same controls can live on the options page and be reached from the extension details menu.

## Implementation Plan

### 1. Add persistent settings

- Define a small settings object for extension preferences.
- Store settings in `chrome.storage.sync` if the values should follow the user across machines, otherwise use `chrome.storage.local`.
- Add safe defaults so existing users get the current behavior until they change it.

### 2. Reverse overlay button order

- Update thumbnail overlay rendering so button order can be built in normal or reversed mode.
- Apply the same setting to player-page buttons if we want the extension to feel consistent across both surfaces.
- Keep the visual layout stable in both modes so the buttons still align correctly in the thumbnail corner and player action row.

### 3. Hide overlays when native buttons exist

- Detect native YouTube controls before attaching custom overlays.
- Skip injection when the native Queue / Watch Later controls are already present, or when YouTube is rendering an equivalent built-in action area.
- Re-check this condition during SPA navigation and mutation-driven rescans so the result stays current as the page changes.

### 4. Add rebuild/reset behavior

- Add a reset action that:
  - removes injected overlays and player buttons from the DOM,
  - clears in-memory markers used to prevent duplicate injection,
  - clears any extension-owned persisted state,
  - re-runs discovery and rebuilds the overlays.
- If we later add any caches in `localStorage`, session storage, or cookies, the reset flow should clear only extension-owned keys and should require explicit confirmation before touching broader site data.
- Reuse the existing observer and route-change flow where possible instead of creating a second discovery loop.

### 5. Add Donate link

- Add a Donate link to the new settings surface.
- Point it to the KoFi page once the URL is known.
- Open the link in a new tab so the user does not lose their current YouTube session.

## Files Likely To Change

- `manifest.json` for the settings surface and any new extension page wiring.
- `src/content.js` for overlay ordering, native-button detection, and rebuild hooks.
- `src/inject.js` if the reset action needs to coordinate with page-context actions.
- `src/styles.css` for any new settings UI and button-order adjustments.
- A new options/popup page script and markup if we choose to add a dedicated settings UI.

## Risks

- YouTube may vary its native button markup across layouts, so native-button detection needs to be conservative and easy to update.
- Clearing browser storage too broadly could break YouTube login/session state, so the reset action must be scoped carefully.
- A settings UI introduces new lifecycle and persistence code, so the first implementation should stay minimal and easy to test.
